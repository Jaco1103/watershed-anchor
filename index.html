What Watershed is

AI projects fail the same way.

The AI generates impressive solutions early, context fragments across sessions, planning never ends or building starts too soon - and the project collapses into a dead-pool: looks like progress, goes nowhere.

Watershed gives the human and the AI one shared structure. The AI's job is not to generate solutions. It is to ask the right questions, in the right order, with enough context to make the answers matter.

Structure before solutions

No solution until the problem, constraints and system type are understood.

Persistent context

One living file is the project's memory. Every session starts by reading it.

Reality arrives early

A prototype forces contact with the real world before the system is over-engineered.

How it actually works

Seven operators. One river.

init-project.md is the living source of truth. Each operator reads it, runs as a focused conversation, and writes its conclusions back. The AI knows the whole map from day one.

0init-project→ 1strategy→ 2archetype→ 3constraints→ 4evaluation→ 5prototype→ 6observability→ 7orchestration→ iteration
Always available, off the chain:reflect

Each operator also writes a project-level slash command, so the next step appears natively in the tool. archetype is where the system, the stack and the data model get decided - the most consequential, and the one I want to harden with you.

Part two · Viah

One AI pass per place.
Every search is deterministic SQL.

Viah builds a walkable, day-by-day itinerary of real places. The architectural bet: Claude tags each place's vibe once (with a confidence score), then every query is a plain SQL scoring function - tag match + proximity + rating. The AI never sits on the query path.

1city live (Copenhagen)
890places
0AI calls per search

Why this shape

Fast, repeatable, and cheap: the same input always returns the same itinerary, and the model cost is paid once at ingest, not per user.

What I want from you

Where this design strains as it scales to many cities, and whether you'd build the matching differently.

Viah · the scoring function

Taste outweighs distance - by design.

Tag match ≤ 60 pts
Proximity ≤ 32 pts
Rating ≤ 8 pts

Tag score = sum of the AI's confidence on matched preferred tags ÷ tags chosen × 60. The enrich-once AI step flows straight into a deterministic number. Before scoring, hard filters gate on quality (rating ≥ 4.0, ≥ 30 reviews), dedup, price, city, walk-radius and a real open-at-hour check.

-- search_places(): the score (sql, STABLE)
COALESCE(
  tag_score_num
  / NULLIF(array_length(p_preferred_slugs,1),0)
  * 60.0, 0.0)                  -- taste
+ CASE WHEN distance_km IS NULL THEN 16.0
    ELSE GREATEST(0, 32.0*(1 - distance_km
                          / p_walk_radius_km))
  END                                -- proximity
+ LEAST(8.0, GREATEST(0,(average_rating-4.0)*8))
                                   -- rating

-- distance: equirectangular on lat/lng (111 km/deg)
-- not PostGIS; no spatial index on places
